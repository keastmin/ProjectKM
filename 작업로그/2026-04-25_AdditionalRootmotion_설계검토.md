# AdditionalRootmotion 설계 검토

## 목적

휴머노이드 애니메이션의 기본 Root Motion을 그대로 믿지 않고, 별도의 `AdditionalRootmotion` 데이터를 통해:

- 원하는 총 이동 거리
- 원하는 이동 방향
- 원하는 구간
- 원하는 속도 변화 곡선

을 정의한 뒤, 매 프레임 계산한 추가 `deltaPosition`을 원본 애니메이션의 `Animator.deltaPosition`에 더해서 최종 이동을 제어하려는 발상에 대한 검토 문서다.

현재 프로젝트 기준 관련 지점은 다음이다.

- [AdditionalRootmotion.cs](/C:/Users/kemin/Unity/ProjectKM/Assets/Scripts/Animation/AdditionalRootmotion.cs)
- [BasicComboAttackState.cs](/C:/Users/kemin/Unity/ProjectKM/Assets/Scripts/Player/SFM/States/BasicComboAttackState.cs)
- [PlayerCore.cs](/C:/Users/kemin/Unity/ProjectKM/Assets/Scripts/Player/PlayerCore.cs)
- [Katana Basic Attack1 Additional.asset](/C:/Users/kemin/Unity/ProjectKM/Assets/Datas/Player/Additional%20Root/Katana%20Basic%20Attack1%20Additional.asset)

## 결론

이 발상은 충분히 적용 가능하다.

특히 현재 프로젝트처럼:

- 공격 상태에서 `OnAnimatorMove()`로 `Animator.deltaPosition`을 누적하고
- `FixedTick()`에서 실제 이동을 적용하며
- 경우에 따라 Motion Warp와 Root Motion을 혼합해서 쓰는 구조

에서는, "기본 애니메이션 이동 + 추가 보정 이동" 계층을 하나 더 두는 방식이 실무적으로 꽤 잘 맞는다.

다만 지금 형태의 `AdditionalRootmotion`은 아직 "아이디어 검증용 최소 구조"에 가깝다. 그대로 확장하면 곧바로 다음 문제가 생긴다.

- 속도 곡선은 만들 수 있지만 총 이동량 보장이 약하다.
- 구간 지정은 가능하지만 정밀 편집성이 낮다.
- 애니메이션 원본 Root Motion과 추가 이동의 책임 분리가 불명확하다.
- Motion Warp와 동시에 쓸 때 우선순위 충돌 가능성이 있다.
- 디자이너가 숫자와 커브만 보고 맞추기엔 작업성이 떨어진다.

즉, 방향성은 맞고, 실제로 쓰려면 "데이터 표현 방식"과 "에디터 도구"를 같이 보강해야 한다.

## 왜 이 방식이 성립하는가

핵심은 `deltaPosition`이 "이번 프레임에 얼마를 이동할지"라는 증분값이라는 점이다.

따라서 공격 애니메이션 재생 중에:

1. 현재 normalized time을 구하고
2. AdditionalRootmotion의 적용 구간 안인지 판정하고
3. 구간 내 진행도 t를 계산하고
4. 커브를 기반으로 "현재 시점의 추가 이동 비중" 또는 "누적 이동 비중"을 얻고
5. 이전 프레임 대비 차이를 구해 추가 `deltaPosition`으로 변환한 뒤
6. 원본 `Animator.deltaPosition`에 더하면

원하는 총 이동량과 속도 프로파일을 별도 데이터로 제어할 수 있다.

즉, 이것은 Root Motion을 대체하는 개념이 아니라, Root Motion 위에 "보정 레이어"를 씌우는 개념으로 이해하면 된다.

## 현재 구조의 장점

### 1. 원본 애니메이션을 다시 손대지 않아도 된다

애니메이션 클립 자체의 루트 이동을 DCC 툴에서 다시 수정하지 않고도, 게임 내 체감 이동거리를 맞출 수 있다.

### 2. 공격마다 원하는 체감 전진량을 따로 맞출 수 있다

같은 카타나 공격이라도:

- 1타는 짧고 날카롭게
- 2타는 조금 더 길게
- 대시 공격은 초반 가속형

처럼 개별 제어가 가능하다.

### 3. 전투 설계와 연결하기 좋다

히트 타이밍, 콤보 입력 가능 시간, Motion Warp 종료 시점처럼 이미 `normalizedTime` 기반으로 관리 중인 시스템과 잘 맞는다.

### 4. Root Motion 완전 제거보다 부자연스러움이 적다

원본 애니메이션의 미세한 몸 움직임은 살리고, 부족하거나 과한 이동만 보정하는 형태이기 때문에 전체를 코드 이동으로 대체하는 것보다 자연스럽게 만들기 쉽다.

## 현재 구조의 핵심 한계

## 1. 커브 value를 직접 속도로 해석하면 총 이동량 오차가 생긴다

지금 자산을 보면 `TargetDeltaPosition`, `TargetStartNormalTime`, `TargetEndNormalTime`, `AdditionalCurve`만 있다.

이때 커브를 "속도"로 볼지 "누적 비율"로 볼지가 매우 중요하다.

속도로 해석하면 문제는 다음과 같다.

- 키 값을 1, 0.5, 0.2처럼 잡아도 총합이 직관적으로 보이지 않는다.
- 프레임레이트와 샘플링 방식에 따라 미세 오차가 누적될 수 있다.
- 디자이너가 "이 값이면 최종적으로 1.8m 간다"를 직관적으로 알기 어렵다.

즉, 커브 모양은 예쁘게 잡았는데 최종 이동거리가 조금씩 어긋나는 일이 반복된다.

## 2. 음수 value와 0 기준 정렬 문제가 있다

속도 커브를 직접 다루면 자연스럽게 이런 문제가 생긴다.

- 감속 곡선을 만들다가 value가 0 아래로 내려감
- 의도하지 않은 역방향 추가 이동 발생
- 클램프를 걸면 곡선 모양이 깨짐
- 0에 맞춘다고 해도 탄젠트 때문에 미세 음수가 생김

즉, "속도 곡선" 자체를 raw 값으로 받으면 안전하지 않다.

## 3. normalized time만으로는 위치 편집 감각이 떨어진다

지금 방식은 디자이너가 숫자와 커브로만 조정해야 한다.

그래서 이런 어려움이 있다.

- 0.17이 실제 어느 자세인지 바로 감이 안 온다.
- "칼을 내리찍기 직전까진 멈추고, 그 다음 반박자에 확 나가고 싶다" 같은 요구를 숫자로 맞추기 어렵다.
- 도프시트처럼 특정 포즈를 보면서 위치를 찍는 방식보다 반복 튜닝 비용이 크다.

## 4. 원본 Root Motion과 추가 Root Motion이 서로 싸울 수 있다

예를 들어 원본 애니메이션이 이미 앞으로 1.2m 가는데 추가 데이터가 또 1.8m를 더하면:

- 총 3m 이동처럼 과해질 수 있고
- 공격 판정과 피격 거리감이 깨질 수 있고
- 타겟 워프와 합쳐졌을 때 돌진이 비정상적으로 커질 수 있다.

즉, "추가"인지 "보정"인지 개념을 분명히 해야 한다.

## 5. Motion Warp와 우선순위 충돌 가능성이 있다

현재 `BasicComboAttackState`는 워프 중에는 Root Motion을 사실상 쓰지 않고, 워프가 끝나면 Root Motion으로 돌아간다.

이 구조에서 AdditionalRootmotion을 넣으면 정책을 먼저 정해야 한다.

- 워프 중에는 AdditionalRootmotion도 무시할지
- 워프 종료 후 남은 구간만 적용할지
- 워프 자체가 AdditionalRootmotion 목표를 덮어쓸지

이 정의가 없으면 실제 체감이 일관되지 않는다.

## 추천하는 보완 방향

## 1. 커브를 "속도"가 아니라 "누적 이동 비율"로 해석하는 방식을 추천

이 문제의 핵심 보완책이다.

`AdditionalCurve.Evaluate(t)`를 현재 시점의 속도로 쓰지 말고, "0~1 사이의 누적 이동 비율"로 쓰는 편이 훨씬 안전하다.

예를 들어:

- t=0 에서 값 0
- t=1 에서 값 1

로 두고, 중간 구간을 ease-out 형태로 만들면:

- 초반에 누적 비율이 빠르게 올라가고
- 후반에 천천히 올라가므로
- 결과적으로 "빠르게 시작해서 점점 감속"한 것과 같은 효과가 나온다.

그리고 실제 프레임 이동은:

- 현재 누적 비율 - 이전 프레임 누적 비율

로 계산하면 된다.

이 방식의 장점:

- 최종 이동량이 항상 `TargetDeltaPosition`과 정확히 일치한다.
- 프레임 오차 누적이 크게 줄어든다.
- 음수 속도 문제를 구조적으로 피하기 쉽다.
- 디자이너가 커브를 "이 시점까지 몇 % 이동했는가"로 이해할 수 있다.

즉, 커브는 "속도 곡선"처럼 보이더라도, 실제 해석은 "누적 분포 곡선"으로 하는 것이 좋다.

## 2. 커브에 제약을 두는 검증 규칙이 필요하다

AdditionalRootmotion 데이터 저장 시 또는 에디터 표시 시 다음 규칙을 강하게 거는 것이 좋다.

- 시작 값은 0으로 스냅
- 끝 값은 1로 스냅
- 중간 값은 0~1 범위로 클램프
- 시간축은 0~1 범위로 클램프
- 곡선은 단조 증가(monotonic)하도록 보정 또는 경고

핵심은 "누적 비율 커브는 감소하면 안 된다"는 점이다.

감소 구간이 있으면 그 구간은 역이동을 뜻하게 되므로, 의도적 특수 케이스가 아니라면 대부분 잘못된 데이터다.

즉, 디자이너가 실수로 음수를 넣거나 뒤로 꺾는 곡선을 만들더라도 시스템이 먼저 막아줘야 한다.

## 3. 데이터 의미를 "추가 이동"보다 "구간 목표 이동"으로 재정의하는 편이 낫다

이름은 `AdditionalRootmotion`이어도 내부 의미는 더 명확해야 한다.

권장 의미:

- 이 구간에서 캐릭터를 최종적으로 얼마만큼 더 이동시키고 싶은가

여기서 "더"의 기준을 명확히 정해야 한다.

실무적으로는 아래 2안 중 하나를 고르는 게 좋다.

### A안. 순수 추가 방식

- 원본 `Animator.deltaPosition`은 그대로 사용
- 여기에 Additional delta만 더함

장점:

- 구현이 단순하다.

단점:

- 원본 Root Motion 영향이 커서 결과 예측이 어렵다.

### B안. 보정 방식

- 특정 구간의 원본 Root Motion 총합을 측정
- 원하는 목표 이동과의 차이만 Additional로 적용

예:

- 원본 구간 이동 0.9m
- 내가 원하는 목표 이동 1.8m
- 실제 Additional 총합은 0.9m만 넣음

장점:

- 최종 결과가 예측 가능하다.
- 애니메이션마다 체감 거리 통일이 쉽다.

단점:

- 구현과 툴링이 조금 더 필요하다.

장기적으로는 B안이 더 낫다.

## 4. 단일 구간/단일 벡터가 아니라 "세그먼트" 구조로 가는 것이 좋다

지금 구조는 한 데이터에:

- 하나의 시작 시간
- 하나의 끝 시간
- 하나의 목표 벡터
- 하나의 커브

만 있다.

하지만 실제 공격 모션은 보통:

- 준비
- 전진
- 멈춤
- 미세 후속 이동

처럼 여러 덩어리로 나뉜다.

그래서 최종적으로는 다음 같은 구조가 더 적합하다.

- 여러 개의 RootMotionSegment 배열
- 각 Segment는 `startNormalizedTime`, `endNormalizedTime`
- 각 Segment는 `localDelta` 또는 `world/forward 기준 이동`
- 각 Segment는 `distributionCurve`

이렇게 하면:

- 0.10~0.22 구간은 급전진
- 0.22~0.35 구간은 정지
- 0.35~0.52 구간은 미세 미끄러짐

처럼 세밀한 설계가 가능하다.

## 5. 방향 정의 기준을 분리해야 한다

`TargetDeltaPosition`은 단순 `Vector3` 하나로 두면 해석이 애매해진다.

이 벡터가:

- 애니메이션 시작 시점의 캐릭터 로컬 기준인지
- 현재 프레임 기준인지
- 월드 기준인지
- 타겟 방향 기준인지

명확하지 않기 때문이다.

공격 이동은 보통 다음 기준 중 하나가 필요하다.

- 캐릭터 시작 로컬 기준 전방
- 현재 바라보는 방향 기준
- 락온 대상 방향 기준

특히 현재 프로젝트는 공격 진입 시 `_targetRot`와 Motion Warp가 있으므로, AdditionalRootmotion도 같은 기준계에 맞춰야 체감이 일관된다.

권장 방식:

- 방향 기준 enum 분리
- `ForwardDistance`, `RightDistance`처럼 평면 기준값으로 분리
- 필요하면 타겟 추적형 옵션 추가

## 6. 애니메이션을 보면서 찍는 문제는 에디터 툴로 해결해야 한다

이건 코드 수학만으로는 해결되지 않는다.

사용자가 지적한 "도프시트처럼 직접 보면서 정의하기 어렵다"는 문제는 맞는 지적이고, 실제로 툴이 없으면 유지보수성이 급격히 떨어진다.

이 문제의 가장 현실적인 보완책은 커스텀 에디터다.

권장 에디터 기능:

- AnimationClip 프리뷰 표시
- 현재 normalized time scrubber
- 특정 normalized time에서 포즈 샘플링
- 구간 start/end 핸들 드래그
- 현재 구간의 예상 누적 이동량 표시
- 씬 뷰에서 이동 궤적 gizmo 표시
- 시작점/중간점/종점 마커 표시
- 원본 Root Motion 궤적과 Additional 적용 후 궤적을 동시에 표시

즉, 숫자를 손으로 맞추는 게 아니라 "보이는 편집"으로 바꿔야 한다.

현재 프로젝트에는 이미 `ComboAttackDataEditor`에 timeline 기반 편집 개념이 들어가 있으므로, 이 패턴을 재사용하면 방향이 좋다.

## 7. "시간 기반" 말고 "거리 기반 확인 도구"도 같이 있어야 한다

디자이너가 진짜 알고 싶은 것은 보통 이 두 가지다.

- 이 공격이 몇 프레임에 얼마나 나가나
- 최종적으로 총 몇 미터 나가나

따라서 에디터나 인스펙터에 최소한 다음 정보는 보여주는 것이 좋다.

- 원본 Root Motion 총 이동 거리
- Additional 총 이동 거리
- 최종 예상 총 이동 거리
- 가장 빠른 순간 속도
- 평균 속도

이 정보가 없으면 커브는 맞췄는데 실제 전투 감각은 계속 어긋난다.

## 질문하신 단점별 직접 답변

## 1. "커브 value를 0 기준으로 맞춰야 하고 음수가 될 수 있다"는 문제

맞다. 그 문제는 실제로 생긴다.

하지만 해결 가능하다. 핵심은 커브를 raw 속도로 쓰지 않는 것이다.

해결책:

- 커브를 누적 이동 비율로 사용
- 시작 0, 끝 1 강제
- 값 범위 0~1 강제
- 감소 구간 금지

그러면 음수 문제, 최종 거리 오차 문제를 거의 동시에 줄일 수 있다.

정리하면:

- "속도 곡선"을 직접 저장하지 말고
- "이 시점까지 전체 이동의 몇 %를 썼는가"를 저장하는 방식이 더 낫다.

## 2. "코드 계산이라 세세한 오차가 존재할 수 있다"는 문제

맞다. 하지만 설계를 잘하면 허용 가능한 수준으로 낮출 수 있다.

오차를 줄이는 핵심은 다음이다.

- 매 프레임 속도 적분 방식보다 누적 비율 차분 방식 사용
- 구간 종료 시 마지막 프레임 보정
- 루프나 트랜지션 중에는 구간 판정을 별도로 안정화
- `normalizedTime`은 `Mathf.Repeat(..., 1f)` 또는 상태 시작 기준 상대값으로 명확히 관리

즉, 오차는 "있을 수밖에 없는 문제"가 아니라, 계산 방식을 잘 고르면 대부분 제어 가능한 문제다.

## 3. "애니메이션을 직접 보면서 찍는 방식이 아니라 정밀 정의가 어렵다"는 문제

이것도 맞다. 그리고 이건 데이터 구조보다 에디터 UX 문제다.

해결책:

- AdditionalRootmotion 전용 커스텀 에디터 제공
- 클립 프리뷰 + 타임라인 + 구간 핸들 제공
- 씬 뷰 궤적 시각화 제공
- 필요하면 "현재 프리뷰 포즈에서 start/end 찍기" 버튼 제공

즉, 이 문제는 "현재 발상이 틀렸다"의 문제가 아니라, "툴 없이 쓰기엔 불편하다"의 문제다.

---

## 내가 권장하는 최종 방향

현 시점에서 가장 추천하는 방향은 아래다.

### 1단계

현재 아이디어를 유지한다.

즉:

- 원본 `Animator.deltaPosition` 기반 이동 유지
- AdditionalRootmotion으로 추가 이동 보정

### 2단계

하지만 데이터 해석은 아래처럼 바꾼다.

- 커브 = 속도 곡선이 아니라 누적 이동 비율 곡선
- `TargetDeltaPosition` = 해당 구간의 최종 목표 추가 이동량
- 실제 프레임 이동 = 현재 누적 비율 - 이전 누적 비율

### 3단계

단일 구간이 아니라 세그먼트 배열 구조로 확장한다.

### 4단계

에디터 프리뷰를 붙인다.

- 타임라인 편집
- 프리뷰 샘플링
- 궤적 시각화

### 5단계

장기적으로는 "추가"보다 "최종 목표 보정" 모델로 발전시킨다.

즉:

- 원본 구간 이동을 샘플링해서 알고
- 원하는 최종 이동과의 차이를 자동 계산해서
- Additional 값을 생성하는 방식

이 단계까지 가면 디자이너는 "최종적으로 1.8m 전진, 초반 급가속 후 감속"만 정하면 되고, 나머지 보정은 시스템이 맡게 된다.

---

## 현재 아이디어에 대한 최종 판단

네 발상은 충분히 적용 가능하고, 방향도 좋다.

오히려 현재 프로젝트 구조에서는:

- Motion Warp
- Root Motion 누적
- 공격별 TimingProfile
- normalizedTime 기반 상태 처리

가 이미 있으므로, AdditionalRootmotion은 자연스럽게 들어갈 수 있는 축에 가깝다.

다만 지금의 단순 데이터 구조 그대로 바로 실전 적용하기보다는, 아래 3가지는 거의 필수 보완이라고 보는 게 맞다.

- 커브를 누적 이동 비율로 해석할 것
- 데이터 검증 규칙을 강하게 둘 것
- 프리뷰 가능한 커스텀 에디터를 만들 것

이 3개만 갖추면, 질문하신 두 가지 핵심 단점:

- 값 오차 문제
- 정밀 편집 문제

를 상당 부분 현실적으로 해결할 수 있다.
